# hi — 产品理念文档

> 状态：草稿 v0.2，持续迭代中
> 来源：作者与 AI 的产品对话记录（2026-05-09）

---

## 一、名字的含义

这个编辑器叫 **`hi`**，不是随意取的，名字本身携带了产品的全部理念：

**`hi` 与 `vi` 的关系**：vi 是 Unix 时代最经典的编辑器，Vim 是"Vi IMproved"，`hi` 延续这条脉络，是面向 AI 时代的全新进化。

**`hi` 与 `ai` 的关系**：字母高度相近，暗示 AI 是这个编辑器的核心基因，不是外挂。

**`hi` 作为命令的优势**：两个字母，极易输入，在命令行里 `hi filename` 随叫随到，比 `vim`、`nano` 都更快打出来。

**`hi` 的人文含义**：打招呼。你打开一个文件，就是在跟它说 hi——触发，打开，开始交互。同时，`hi` 是人类最友好的问候方式，体现了 AI 应有的态度：智能、但亲切，强大、但不傲慢。

---

## 二、这个产品为谁而做

**为软件开发工程师打造的、以 AI 为核心交互范式的 terminal 文本编辑器。**

目标用户画像：

- 有代码开发经验（Java、Python 等），理解软件架构和原理
- 日常在 terminal 环境工作，需要快速编辑配置文件、查看日志、系统运维操作
- 用过 Vim，但被 Vim 的核心问题困扰：**能力强大但不可发现**——操作组合必须先记住才能用，而不是用的时候能找到
- **不是**大型 IDE 的替代品，定位是 terminal 里的快速、轻量、智能编辑器

典型使用场景：

- 快速打开配置文件（XML、YAML、JSON、TOML、.env 等）进行编辑
- 系统运维时进入目录快速浏览文件结构
- Debug 时查看、搜索日志文件
- 临时的文本处理（替换、格式转换等复杂操作）

---

## 三、核心问题与设计原点

Vim 最大的问题不是功能不够强，而是**操作能力不可发现**。用户必须先花大量时间背快捷键，才能解锁那些强大的能力。大多数人在"记不住"和"不知道有这个操作"之间反复受挫，最终放弃。

**hi 的设计原点**：让编辑器的能力变得可发现、可触达，用自然语言消除"我不知道怎么做"的障碍。

## 四、模式设计

保留 Vim 经过验证的模式理念，但重新定义各模式的职责：

### 基础模式（继承自 Vim 核心理念）

| 模式 | 触发键 | 退出 | 职责 |
|---|---|---|---|
| Normal 模式 | 默认启动 / `Esc` | — | 导航、操作文本，不直接输入字符 |
| Insert 模式 | `i` | `Esc` | 直接输入文本 |
| Visual 模式 | `v` / `V` | `Esc` | 选择文本块 |
| Command 模式 | `:` | `Esc` / `Enter` | 执行 Ex 命令 |
| Search 模式 | `/` | `Esc` / `Enter` | 文本搜索（向前），`?` 历史上是向后搜索，此处重新定义 |

### AI 模式（新增核心模式）

| 触发键 | 职责 |
|---|---|
| `?` | 进入 AI 交互入口，AI 根据意图复杂度自动选择响应形式 |

`?` 的选择理由：语义最贴切，问号天然代表"提问"；与 `/`（文本搜索）形成自然对称——`/` 搜文本，`?` 问 AI；按键位置相邻，符合直觉。

## 五、AI 交互设计——最核心的创新

### 5.1 设计原则：AI 不主动介入

AI **绝对不主动打扰**。用户没有明确按下 `?` 之前，AI 不发出任何提示、不主动建议、不中断用户操作。这是对用户注意力的基本尊重。

### 5.2 自适应响应：AI 自己判断复杂度

按下 `?` 输入自然语言后，AI 分析用户意图，**自动**决定响应形式，用户无需手动切换：

```
简单意图（单条命令可完成）
  → 幽灵文字模式：AI 直接把命令"预填"到命令行
    - Tab：确认并执行
    - 继续打字：幽灵文字消失，回到用户自己输入

复杂意图（需要多步操作）
  → 执行规划模式：AI 展示多步操作计划
    - 用户确认后按步执行
    - 未来支持 yolo 模式（跳过确认直接执行，参考 Claude Code）

提问/顾问意图（用户问"怎么做"）
  → 顾问模式：AI 给出文字解释和按键提示，不执行任何操作
```

### 5.3 幽灵文字模式（简单意图示例）

用户按 `?`，输入：`把所有"中国"替换为"法国"`

AI 判断这是单条替换，直接在命令行预填（灰色幽灵文字）：

```
:%s/中国/法国/g█
```

状态栏提示：`[Tab] 确认执行    [继续输入] 取消建议`

用户按 Tab → 执行，完成。

### 5.4 执行规划模式（复杂意图示例）

用户按 `?`，输入：`把所有数字换成中文`

AI 判断需要多步骤，弹出规划面板：

```
╭─ AI 执行计划 ──────────────────────────────────────╮
│ 意图：将文件中所有阿拉伯数字替换为对应的中文数字      │
│                                                    │
│ 步骤 1  匹配所有数字（正则 \d+）                    │
│ 步骤 2  按映射表替换（1→一，2→二，...，10→十）       │
│ 步骤 3  处理多位数（11→十一，100→一百，以此类推）     │
╰────────────────────────────────────────────────────╯
[y] 确认执行    [n] 取消    [e] 编辑计划
```

### 5.5 顾问模式（用户问"怎么做"时）

用户按 `?`，输入：`怎么选中当前段落`

AI 识别这是在询问操作方法，给出提示，**不执行任何操作**：

```
[AI] 选中当前段落：Normal 模式下按 vip
     v  → 进入 Visual 模式
     ip → 选中 inner paragraph（当前段落内容）
```

### 5.6 未来：yolo 模式

在配置文件中启用 `yolo_mode = true` 后，执行规划模式跳过用户确认步骤，AI 直接执行。适合熟悉产品、信任 AI 判断的用户。

## 六、可发现性设计——常驻底部状态栏

**核心理念**：永远不让用户陷入"不知道下一步能做什么"的困境。

屏幕最底部始终保留一行提示区，根据当前**模式**和**上下文**动态预测并显示最可能用到的按键，像 IDE 智能补全一样有预测性：

```
Normal 模式（光标在普通文本行）：
[i]插入  [v]选择  [dd]删行  [yy]复制行  [p]粘贴  [u]撤销  [?]AI  [gg/G]首/尾

Visual 模式（已选中多行）：
[y]复制  [d]删除  [>]缩进  [<]反缩进  [?]AI操作选区  [Esc]退出选择

Normal 模式（光标在文件路径上）：
[gf]打开文件  [yiw]复制路径  [ciw]替换  [?]AI  [Esc]
```

**设计原则：**

- 以**具体按键**为主，看一眼就能立刻使用，零额外学习成本
- 有**预测性**，根据光标位置、文件类型、选中内容，推断用户最可能的下一步操作
- 内容**动态变化**，不是静态帮助文档

## 七、文件树

内置文件树，功能对齐 Vim NERDTree：

- `hi .` 打开当前目录，左侧展开文件树面板；编辑文件时也可用快捷键呼出
- `hjkl` 或方向键导航，`Enter` 打开文件
- 支持：新建、删除、重命名、复制
- 文件树与编辑区之间通过快捷键切换焦点，全程无需鼠标

## 九、配置文件

路径：`~/.hirc`
格式：TOML

```toml
# ~/.hirc — hi 编辑器配置文件

[general]
# 启动时默认显示行号
line_numbers = true
# tab 宽度
tab_width = 4

[ai]
# LLM 后端，支持任何兼容 OpenAI API 格式的服务
api_base_url = "https://api.openai.com/v1"
api_key = ""           # 也可通过环境变量 HI_API_KEY 设置
model = "gpt-4o"
# yolo 模式：跳过 AI 执行计划的用户确认步骤
yolo_mode = false

[theme]
colorscheme = "default"
```

LLM 后端设计为**完全可配置**，通过 `api_base_url` + `model` 组合支持：
- OpenAI（`api.openai.com`）
- Anthropic Claude（通过兼容层）
- 本地 Ollama（`localhost:11434`）
- 任何兼容 OpenAI API 格式的服务

## 十、产品边界与定位

### 是什么
- Terminal 原生，无 GUI 依赖，无鼠标操作
- 轻量快速，随叫随到（目标启动时间 < 100ms）
- AI 是一等公民，内置于核心，不是外挂插件
- 覆盖单文件编辑 + 目录浏览，够用即止

### 不是什么
- 不是 VS Code / JetBrains 的 terminal 替代品
- 不是 IDE（不内置调试器、构建系统）
- 不追求完整兼容 Vim / Neovim 的插件生态

### 与 Vim / Neovim 的核心差异

| 维度 | Vim / Neovim | hi |
|---|---|---|
| 能力发现方式 | 背文档、背快捷键 | 状态栏实时预测提示 |
| AI 集成 | 第三方插件，体验割裂 | 原生内置，一等公民 |
| 复杂操作 | 手动组合命令序列 | 自然语言描述，AI 自动规划执行 |
| AI 触发 | 无统一入口 | `?` 键，专属且直觉化 |
| 学习曲线 | 陡峭，记忆成本高 | 渐进式，状态栏随时提示 |
| 定位 | 全能编辑器 | 快速、智能的运维/配置场景编辑器 |

### 多光标编辑（Multi-cursor）

> 状态：**待研究，暂不实现**（2026-05-09 记录）

**是什么**：在文档的多个位置同时放置光标，所有光标同步接收输入，实现批量编辑。VS Code（`Ctrl+D` 选中下一个相同词）、Sublime Text（`Cmd+Click` 添加光标）是典型实现。

**为什么有价值**：
- 批量重命名变量（比宏更直观，所见即所得）
- 同时编辑多行相似结构（比正则替换更精确）
- 视觉反馈即时，操作结果一目了然

**实现挑战**：
- 需要维护多个光标状态（位置、选区、模式）
- 光标之间可能产生冲突（两个光标操作同一行时的合并策略）
- TUI 环境下渲染多个光标位置，需要特殊处理
- Vim 的 modal editing 模型与多光标的交互语义需要重新设计

**与宏的关系**：多光标和宏解决的是同类问题（批量重复操作），但交互范式不同。多光标是"同时可见、同步操作"，宏是"先录制、后回放"。两者可以并存，适用不同场景。

**后续研究方向**：参考 Helix 编辑器的多光标实现（Helix 原生支持多光标，且与 modal editing 结合得较好）；研究 `crossterm` 在 TUI 中渲染多光标的可行方案。
