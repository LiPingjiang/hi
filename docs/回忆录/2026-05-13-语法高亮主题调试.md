# 回忆录 · 2026-05-13 · 语法高亮主题调试

---

## 背景

用 `hi` 编辑器打开 `~/.openclaw/openclaw.json` 时，发现 JSON 文件的字符颜色很暗，几乎看不清。排查后确认这是 base16-ocean.dark 主题的特性：JSON key 和 string value 都被渲染为同一个暗绿色 `#A3BE8C`，在深色背景 `#2B303B` 上对比度不足。

## 问题诊断

写了一个测试遍历 syntect 内置的全部 7 个主题，dump 每个主题对 JSON 的颜色分配：

```
base16-ocean.dark:   key=#A3BE8C  value=#A3BE8C  num=#D08770  bool=#D08770
base16-eighties.dark: key=#99CC99  value=#99CC99  num=#F99157  bool=#F99157
base16-mocha.dark:   key=#BEB55B  value=#BEB55B  num=#D28B71  bool=#D28B71
Solarized (dark):    key=#2AA198  value=#2AA198  num=#6C71C4  bool=#B58900
InspiredGitHub:      key=#183691  value=#323232  num=#0086B3  bool=#0086B3  ← 唯一区分 key/value 的，但是亮色主题
```

结论：所有 base16 系暗色主题都有同样的问题。根本原因是 JSON 的 key 和 value 在 TextMate 语法中都被标记为 `string.quoted`，只有少数主题会针对 `meta.structure.dictionary.key` 和 `meta.structure.dictionary.value` 设置不同的 scope 规则。

## 尝试的方案：自定义 hi-dark 主题

创建了一个基于 Monokai 色板的 `.tmTheme` 文件（`themes/hi-dark.tmTheme`），专门为 JSON key/value 设置了不同的 scope 规则：

- JSON key → `#66D9EF`（亮青色）
- JSON string value → `#E6DB74`（金黄色）
- 数字/布尔/null → `#AE81FF`（紫色）
- 结构字符 → `#F8F8F2`（亮白）

技术实现：

1. 在 `Cargo.toml` 添加 syntect 的 `plist-load` feature（引入 plist 解析能力）
2. 用 `include_bytes!` 在编译时嵌入 `.tmTheme` 文件
3. 在 `SyntectHighlighter::new` 和 `MdRenderer::new` 中用 `ThemeSet::load_from_reader` 注册自定义主题
4. 将默认主题从 `"base16-ocean.dark"` 改为 `"hi-dark"`

## 验证方式：xterm.js 浏览器模拟

由于 CatDesk 沙箱环境禁止 PTY 操作（`openpty: Operation not permitted`），无法在真实终端中运行 `hi` 编辑器。采用了以下替代方案：

1. Rust 测试生成 ANSI 转义序列帧（`.ans` 文件）
2. 用 xterm.js 在浏览器中渲染这些帧
3. 通过 `catdesk browser-action screenshot` 截图
4. AI 视觉分析截图效果

在浏览器截图中，新主题的 JSON 颜色区分确实很明显。

## 为什么撤回

**浏览器模拟与真实终端的渲染差异太大，用 xterm.js 做视觉验证不够严谨。**

具体问题：

1. **颜色映射不一致**：xterm.js 的 16 色 palette 映射、RGB 颜色渲染、gamma 校正等与真实终端（iTerm2、Terminal.app、Alacritty 等）存在差异。在浏览器里看着对比度很好的颜色，在真实终端里可能完全不同。

2. **字体渲染差异**：浏览器的字体渲染（subpixel antialiasing、hinting）与终端模拟器不同，影响文字的视觉清晰度和颜色感知。

3. **背景混合差异**：终端模拟器对 ANSI 颜色的背景混合、透明度处理各不相同，xterm.js 无法代表所有终端的行为。

4. **无法验证真实用户体验**：最终用户是在真实终端里使用 `hi`，不是在浏览器里。如果验证环境和使用环境不一致，测试结果没有意义。

正确的做法应该是参考 glow 等成熟项目的语法高亮方案，在真实终端环境中调试和验证。

## 保留的收获

虽然主题改动被撤回，但这次调试过程中建立的基础设施仍然有价值：

1. **颜色 dump 方法论**：通过 `SyntectHighlighter` 遍历主题、dump 每个 token 的颜色分配，可以快速评估主题对特定语言的表现。
2. **syntect 自定义主题加载路径**：确认了 `plist-load` feature + `ThemeSet::load_from_reader` + `include_bytes!` 的技术路线是可行的，未来需要时可以直接复用。
3. **JSON scope 规则**：确认了 `meta.structure.dictionary.key string` 和 `meta.structure.dictionary.value string` 是区分 JSON key/value 的正确 TextMate scope。

## 后续方向

1. **参考 glow 的实现**：glow 使用 glamour 库做终端 Markdown 渲染，其语法高亮方案值得研究。
2. **真实终端验证**：需要在非沙箱环境中（真实 macOS 终端）运行 `hi` 编辑器，用肉眼或 `screencapture` 截图来验证效果。
3. **主题选择策略**：可能需要根据终端的背景色自动选择合适的主题，而不是硬编码一个默认值。
